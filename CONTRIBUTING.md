# Contributing

## 開発環境

```bash
npm ci
npm run dev
```

本番ビルド確認：

```bash
npm run build
```

依存関係を追加・更新した場合は`package-lock.json`もコミットする。通常のCI・レビュー確認では`npm ci`を使用する。

## ブランチ

- `main`：公開ブランチ
- `docs/<topic>`：文書追加・修正
- `feat/<topic>`：サイト機能追加
- `fix/<topic>`：不具合修正

1 Issue、1 Branch、1 Pull Requestを原則とする。

## ページ構成

各実践ページには可能な範囲で次を含める。

1. このページで分かること
2. 前提環境
3. 背景と仕組み
4. 実行手順
5. 期待される結果
6. 内部で起きていること
7. よくあるエラー
8. 次に読むページ
9. 参考資料

## 検証日の更新

全ページのfrontmatterに次を記載する。

```yaml
last_verified: 2026-07-11
```

- 形式は`YYYY-MM-DD`
- コマンド、対象OS、ROS 2ディストリビューション、外部リンクを確認した日を記録
- 誤字修正だけの場合は更新不要
- 手順や前提環境を変更した場合は同じPull Requestで更新

## 表記

- 製品名は`ROS 2`と表記
- コマンドには対象環境を明記
- 未検証手順を検証済みと表現しない
- 公式文章を転載せず、要約と参照リンクを使用
